- ~~PII redaction in DEBUG probe~~ (`1.5.3-beta.2`) — the value-rich format above turned SETTINGS sub-keys that carry text (device name `0x09`, user name `0x01`, email `0x02`, phone `0x03`) into trivially decodable hex. `_redact_if_text` masks any value whose bytes are ≥ 3 long and entirely printable ASCII as `<text:Nb>` (length preserved, content masked). Numeric readings keep their full hex value because they always include at least one non-printable byte. UTF-8 multi-byte text and ≤ 2-byte strings are acceptable partial leaks. Hardens the workflow before asking any user to paste a DEBUG capture publicly. (#181)
- ~~Per-device HTS deltas no longer mis-route to hub-network parser~~ (`1.5.3-beta.3`) — `_is_network_state_delta` was over-matching: it ran `_extract_direct_kv(params[1:])` against every non-body `UPDATES` message and any kv dict it produced was tested against `KEY_HUB_POWERED` (`0x03`) and four other hub-level keys. For a per-device payload `[sub_key, device_id_4b, k, v, k, v, …]` `_extract_direct_kv` happily skipped the 4-byte device_id and started alternating from there — and the operational state byte `0x03` is so common as a *value* in Ajax device payloads that the alternation lined `0x03` up as a kv-dict KEY almost every time, firing the heuristic and routing the delta to the hub-network parser. The bug was completely silent: per-device live updates went into the wrong parser, `device_readings` was only ever seeded by the initial STATUS_BODY, and `RestoreSensor` made the stale values look "live" after every restart. The router now classifies by payload shape (a 4-byte `params[1]` means per-device) before running the heuristic. Affected every electrical-measuring device, not just the Outlet that exposed it. Discovered while triaging @SaetanSaDiablo's #179. (#184)

## Known Limitations

These are protocol-level limitations that cannot be resolved:

- **Hub tamper (lid) real state** — status exists in proto but server doesn't send it in `StreamLightDevices`
- **Photo on-demand URL retrieval** — v2 capture works but photo URL via v3 detection area stream returns `permission_denied`
- **SpaceControl keyfob listing** — keyfobs don't appear in `StreamLightDevices`
- **Motion detection when disarmed** — Ajax firmware disables motion reporting when disarmed (battery conservation)
- **Shock/vibration as persistent sensor** — these are alarm events, not persistent statuses
